严禁使用 git worktree. 当前分支与你收到任务的分支不同时，你应该使用 git stash 确保工作区干净后切换过去开始任务，并在总结时告知用户。

落盘不光是要写入文件，而且要将一整个任务的修改提交为 commit，用户将以 commit 为单位决定批准还是回退.

commit 后就 push 到 remote，如果遇到网络问题就先跳过并告知用户。

遇到需要 sudo 的任务时，不要因为命令里有 sudo 就把整段流程都留给用户做；应先执行所有不需要提权、可由你完成的准备、下载、配置、构建和验证步骤。只有真正需要用户输入 sudo 密码或写入系统目录的步骤，才用可用的终端程序拉起一个交互式终端，把 sudo 命令交给用户完成；终端退出后继续后续验证和流程。

如果这轮用户直接提 bug，不要先修 bug 再做测试，这样无法确定测试是否能真的反应 bug，应该先找到稳定复现路径并落盘为测例、再进行修复、回归循环直至确认 bug 修复，再落盘。

严禁未经用户明确要求，就擅自加入所谓“兼容”“兜底”“自动纠正”“回退路径”代码。

如果一个输入本来就应该报错、404、失败或暴露真实问题，就保持原语义，不要为了“看起来更稳”偷偷改成重定向、fallback、猜测、别名映射或静默吞错。

只有在用户明确要求兼容策略，或有已落盘的测试/需求明确规定必须兼容时，才能加入兼容层；加入前要先说明触发条件、边界和副作用，避免用模糊的“更健壮”当理由。
